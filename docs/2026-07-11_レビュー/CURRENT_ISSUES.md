## P1: 品質担保の欠落 — 訴求と検証のギャップ

### 1. 生成品質の eval が存在しない

ガードレール（捏造禁止・1機能1CEP・DBA 維持）は**プロンプトのソフト強制のみ**で、
これは ADR-007 が明示的に受け入れたトレードオフだが、その担保である HIL は「人が毎回
気づける」ことを前提にしている。プロンプト変更・モデル更新（現在は `gemini-3.5-flash` 主体、
行単位抽出のみ `gemini-3.1-flash-lite`）で静かに劣化しても検知できない。ADR-016 で個別対応が
3ゲート＋人数分岐（個別方式／セグメント方式）に広がり、生成経路が増えたぶん回帰面も広がった。
**対処**: ROADMAP S-3 の eval ハーネスを最優先で導入し、CI に組み込む。

### 2. HIL の承認がプロンプト規律に依存している

`run_assembly`（全件確定）・`generate_individual_deliverables`（個別方式の確定）は「明示承認必須」
だが、それを強制するのはシステムプロンプトの指示のみで、**構造的な強制（承認トークンなしで
ツールが実行を拒否する等）がない**。ADR-016 も「サーバー側の承認強制は本 ADR のスコープ外」と
明記しており、この負債は据え置かれたままである。テナント分離では「規律ではなく構造で」を
貫いたEventTune の思想からすると、ここだけ規律頼みなのは一貫性を欠く。**対処**: 確定系ツール
（`run_assembly` / `generate_individual_deliverables`、将来の送信系）にサーバー側の承認フラグ
（直前の承認イベントを検証してから実行）を導入する。

### 3. SSE イベント処理のパーサにテストがない

7/2 に指摘した「フロントエンドのテスト 0 件」は vitest 導入で解消したが、テスト対象は
ユーティリティ・API クライアント・データテーブル系であり、**最も壊れやすい SSE イベント →
UI 状態のパーサ**（`code` / `code_result` / `tool_result` 等の解釈、`frontend/app/(app)/agent/page.tsx`、
1,200 行超）は依然テストの外にある。**対処**: SSE イベント → UI 状態の変換部分を関数として
切り出し、そこだけでもユニットテストを付ける（全面的な E2E より費用対効果が高い）。

## P2: セキュリティ・運用（条件付きリスク）

### 1. アップロードファイル経由のプロンプトインジェクション

取り込みパイプライン（ADR-015 の統一パイプライン）はファイル内容（CSV セル・テキスト）を
AI プロンプトに埋め込む（バッチ横断理解・ドキュメント抽出・行単位抽出）。悪意ある CSV に
「これまでの指示を無視して…」が仕込まれた場合の影響評価がされていない。
**緩和要因**: AI ツールは closure 束縛で自スペース外に到達できず、取り込み AI の出力は
構造化スキーマ経由で決定論 Python が検証するため、被害半径は構造的に限定されている。
**対処**: 脅威として文書化した上で、抽出結果の異常値検知（例: appeal_summary への
指示文混入チェック）を eval に含める。マルチテナント SaaS を名乗る以上、
「評価済みで受容したリスク」と「未評価」の差は大きい。

### 2. Cloud Run BackgroundTask のスケールイン耐性

取り込み（`POST /api/integration/batches`）は BackgroundTask で実行される。Cloud Run の
インスタンス縮退・再起動でジョブが黙って死ぬと、`IntegrationJob` が進行中のまま残る。
**対処**: 当面はジョブのタイムアウト検知（一定時間更新のないジョブを failed 化）で足りる。
根本対応（Cloud Tasks）は ADR-002 の移行トリガー発火時でよい。

### 3. サンドボックス 14 日失効の UX

Agent Engine サンドボックスは 14 日無使用で消える（ADR-009 記載）。現状はエラーを返して
`get_space_data` 再実行を促す設計だが、この回復がユーザー体験としてどう見えるか
（チャットにエラーが出るのか、自動回復するのか）の確認・改善余地がある。
**対処**: `run_python_code` 内で失効を検知したら自動で再作成＋CSV 再投入して
リトライする（ユーザーに見せない）。

### 4. 総当たりコサインのスケール限界

設計上の意図的選択（ADR-008/010）であり現時点で問題ではないが、**再評価トリガー
（スペースあたり件数の閾値）が数値として定義されていない**。「大規模化したら」では
監視できない。**対処**: 例えば「1スペース 5,000 persons で計測し、分類レイテンシ p95 が
X 秒を超えたら近似最近傍へ」のように、トリガーを測定可能な形で ADR に追記する。

  戻らない。原則5（Historical Intelligence）の完成を妨げる最大の構造的ギャップであり、

- **課金が未接続**: metering / plans は実装済みだが Stripe・プランゲーティングが未配線。
  クレジット超過時の挙動が未定義のため、現状は実質無制限（ROADMAP M-5）。
- **オンボーディングの前提知識**: 現体験は「何をチャットで頼めるか」をユーザーが
  知っている前提。初回スペース作成後の空状態で、サンプルデータ投入や指示例の提示が
  ないと、プロダクトの価値に到達する前に離脱しうる。デモでは問題にならないが、
  実利用では最初の壁になる。[CAPABILITY_MAP.md](CAPABILITY_MAP.md)（得意・不得意・
  頼み方）はこの穴を埋めるオンボーディング素材の原型として使える。

## P2: ポジショニング起点の課題（市場検討で顕在化）

[POSITIONING.md](POSITIONING.md) の市場分析から、プロダクト側に返る課題が3点見えた。

- **リポジトリ名がカテゴリ誤認を招く**: プロダクト名は **EventTune** に確定（ADR-014）したが、
  リポジトリ名 `marketing-mail-generator` は依然「AI メール生成ツール」——市場で最も
  コモディティ化したカテゴリ——を自称する名前のまま。命名規約（PHILOSOPHY_AND_NAMING §6）が
  `generator` をシステム名に使うことを明示的に禁じているのに、リポジトリ名自体が違反している
  状態は残る。GitHub リネームは旧 URL から自動リダイレクトされコストが低い
  （App Hosting / WIF 連携の追従確認は要、ADR-012）。→ 名称確定の経緯は
  [NAMING_PROPOSAL.md](NAMING_PROPOSAL.md)。
- **差別化の核ほど露出が弱い**: 「AI がフォローメールを書く」は Cvent 等も到達済みで、
  もはや競争軸ではない。EventTune の差別化（取り込み根拠・自由分析・イベント横断
  蓄積）のうち、**取り込み根拠の露出は ADR-017 で「由来を追う」機能をいったん撤去**した
  ため、現状は成果物側の根拠のみが見える状態。分析はチャート未対応（ROADMAP S-4）。
  取り込み側の透明性 UI は再設計待ち（S-1）で、**見せ場側の実装が薄い**構図は続いている。
- **「向いていないこと」の線引きが利用者に見えない**: なんでも頼めるチャット UI は、
  Web リサーチや汎用ライティングなど本領外の依頼も受け付けてしまい、期待外れの体験を
  生みうる。CAPABILITY_MAP の内容をエージェントの応答（本領外の依頼への正直な案内）と
  オンボーディングに反映する実装が必要。

## 総括 — 直すべき順番

1. **ドキュメント同期の仕上げ**（P1）: セッションサービス記述（`InMemorySessionService`）の
   是正と PM.md の整理。残り 1 箇所だが技術訴求の核に直結する
2. **eval ハーネス**（P1）: 品質主張を検証可能にし、以降の変更の安全網とする。個別/セグメント
   両経路（ADR-016）に対応させる
3. **HIL の構造的強制**（P1）: 確定系ツール（`run_assembly` / `generate_individual_deliverables`）に
   サーバー側承認検証を入れ、思想の一貫性を回復する
4. **SSE パーサのテスト**（P1）: 最も壊れやすいプロトコル解釈に絞ってユニットテストを付ける
5. **P2 群**: トリガーの数値化と文書化を先に行い、実装はトリガー発火まで待つ

EventTune の強みは「規律ではなく構造」「記録された意思決定」にある。上記の課題は
いずれも致命傷ではなく、**その強みを自分自身に適用しきれていない箇所**の列挙である。
逆に言えば、直し方はすべて既存の設計原則の中に書いてある。7/2 から 1 週間で P0 を含む
複数を実際に消し込めたこと自体が、その運用が機能している証左でもある。
