# 戦略スコープ — 何を実施し、何を実施しないか

[POSITIONING.md](POSITIONING.md) の帰結として、EventTune は **業務全体を置き換えるものではない**。
HubSpot（MA/CRM）の代替でも、Cvent や EventHub（イベント運営）の代替でもなく、
**他のプロダクトができない部分を担い、他のプロダクトと連携して業務を完遂する**存在である。
そうであるなら、「どこまでやるか」は成り行きではなく戦略として決めなければならない。
本ドキュメントはその線引き——**Own（自分でやる）/ Integrate（つないで任せる）/
Don't（絶対にやらない）**——を定義する。

---

## 1. フレーム — System of Record ではなく System of Intelligence

BtoB マーケティングのソフトウェアは、大きく2種類に分かれる。

- **System of Record（SoR・記録のシステム）**: 顧客レコードの正本・商談パイプライン・
  配信基盤を持つ。CRM / MA がこれであり、企業はすでに投資済みで、置き換えの摩擦は極めて高い。
- **System of Intelligence（SoI・判断のシステム）**: SoR の上（あるいは手前）で、
  散在するデータを意味に変え、判断を作り、根拠を残す。

**EventTune は SoI であり、SoR にならない。** これが全スコープ判断の第一原理である。
オントロジー（Person / Event / …）はレコードの「正本」ではなく、判断のための
**意味の写像**——正本は CRM にあってよいし、なくてもよい。

業務フロー全体に重ねると、担当領域はこうなる:

```
イベント企画 ─▶ 集客 ─▶ 開催・リード獲得 ─▶ データ整理 ─▶ 判断・施策設計 ─▶ 配信・実行 ─▶ 商談 ─▶ 受注
   (Don't)   (Don't)   (Integrate:      ╔════════════ Own ════════════╗   (Integrate:    (Integrate:
                        名刺/リード      ║  意味統合 → 分析・セグメント   ║    MA・配信       CRM へ渡す)
                        ツールから       ║  設計 → 根拠つき成果物 →     ║    ツールへ)
                        受け取る)        ║  蓄積・学習（イベント横断）    ║        │
                                        ╚══════════════▲══════════════╝        │
                                                       └── 効果データを受け取る ──┘
                                                           (Integrate: 学習ループ)
```

## 2. 判断基準 — 4つの問い

個別の機能要望を Own / Integrate / Don't に振り分ける基準:

| # | 問い | Yes なら |
|---|---|---|
| Q1 | **オントロジーと判断に直結するか？** — その機能はデータの意味・セグメント・根拠・蓄積を豊かにするか | Own に傾く |
| Q2 | **すでにコモディティ化しているか？** — 成熟した専業ツールが複数あり、差別化の余地がないか | Integrate / Don't に傾く |
| Q3 | **規制・運用の重荷を背負うか？** — メール到達性・購読解除管理（特定電子メール法 / CAN-SPAM）・個人情報の正本管理など、継続的コンプライアンス運用が発生するか | Don't に傾く |
| Q4 | **蓄積が複利になるか？** — 使うほどデータ資産が増え、スイッチングコストになるか | Own に傾く |

例: 「メールを送りたい」という要望は、Q1=No（送信自体は意味を生まない）・Q2=Yes・Q3=Yes・
Q4=No（送信そのものは蓄積しない。ただし**送信結果は** Q1/Q4=Yes）。
→ 送信は Integrate、結果の取り込みは Own。線引きが機械的に導ける。

## 3. Own — 自分でやる（ここで勝つ）

| 領域 | 内容 | 判断根拠 |
|---|---|---|
| **意味統合** | 散在ファイル（リスト・アンケート・概要・費用）の取り込み・名寄せ・オントロジー化 | Q1・Q4 とも最大。市場の空白（POSITIONING §2）。競合が構造的に持てない |
| **判断** | セグメント軸の設計・分類・意味検索（この人に合うもの）・自由分析 | Q1。プロダクトの知能の中核 |
| **根拠つき成果物** | 個別化された成果物（メール文面・トークスクリプト・提案書）の生成と、全判断の根拠記録 | Q1。Auditable AI は調達要件レベルの差別化 |
| **蓄積・学習** | イベント横断の実績・関心・（将来）施策結果の蓄積と、それを使う判断の改善 | Q4 の複利そのもの。防御性の源泉 |
| **業務文脈の統治** | テナント分離・HIL 承認・ブランドガードレール・使用量計測 | SaaS としての信頼の土台。外部化できない |

## 4. Integrate — つないで任せる（境界の設計）

連携は「あれば便利」ではなく、**共存ポジション（POSITIONING §5）の実装**である。
各連携の方向とデータを設計として明示する:

| 連携先 | 方向 | 受け渡すもの | 備考 |
|---|---|---|---|
| 名刺・リードキャプチャ（Sansan・Cvent/iCapture・Blinq 等） | **In** | リード・接客記録 | 現状はファイル経由で実現済み。API 連携は摩擦削減（ROADMAP M-3） |
| CRM / MA（HubSpot・Salesforce・国産 MA） | **In** | ハウスリスト・企業属性 | 「整える前」はこちら、正本は先方 |
| 配信ツール / MA | **Out** | セグメント＋成果物（宛先・文面・根拠） | **現状の CSV エクスポートは既にこの思想**。コネクタ化で摩擦を減らす（M-1/M-3） |
| CRM / SFA | **Out** | ホットリード・推奨アクション・根拠 | 商談管理は先方。こちらは「誰に・なぜ」を渡す |
| 配信ツール | **In** | 開封・クリック・返信などの結果 | **学習ループの生命線**。結果はファクト（例: `deliverable_outcomes`）としてオントロジーに Own する（M-1） |

連携の原則: **データの意味と判断はこちらに残し、実行と正本は先方に任せる。**
取り込んだものは必ずオントロジーを通す（原則2）ので、コネクタが増えても
アーキテクチャは変わらない——増えるのは取り込みソースと書き出し先だけである。

## 5. Don't — 実施しない（明文化された「ノー」）

| やらないこと | 判断根拠 | 代わりに |
|---|---|---|
| **メール配信基盤**（SMTP・到達性・IP レピュテーション・購読解除管理） | Q2 コモディティ × Q3 規制の重荷（特電法/CAN-SPAM 運用）。1行も意味を生まない | 配信は SendGrid / MA へ Out 連携。**M-1「送信連携」はコネクタと結果取り込みまでであり、配信基盤の内製は含まない** |
| **CRM / SFA**（顧客正本・商談パイプライン・活動管理） | SoR は既存投資の置き換え戦争。Q2 | ホットリードと根拠を Out で渡す |
| **イベント運営ロジ**（申込ページ・受付・当日運営・配信プラットフォーム） | Q2（Cvent/EventHub の本丸）。開催「前」は判断資産に乏しい | 開催後のデータを In で受け取る |
| **Web / LP ビルダー・広告運用** | Q1=No（オントロジー外）。CAPABILITY_MAP §2「向いていない」と同根 | — |
| **汎用コピーライティング**（接地なしのブログ・SNS 等） | Q1=No。汎用 AI・ライティングツールの土俵 | 顧客はそれらと併用すればよい |
| **名刺 OCR・スキャン** | Q2（専業が成熟）。読み取りは意味を生まず、意味化はその後の工程 | 名刺ツールの出力を In で受け取る |

**Don't の効用**: 「やらない」が明文化されていると、①ロードマップの議論が「やるか
やらないか」から「どの Own/Integrate を先にやるか」に絞られる、②営業・PMM が
「HubSpot と競合しますか？」に迷いなく答えられる（→ しません。連携先です）、
③開発リソースが規制運用やコモディティ再発明に流出しない。

なお唯一の例外条件も定めておく: **Don't 領域を再検討するのは、その領域がEventTune の
学習ループのボトルネックになったときだけ**（例: 配信ツール側から結果データがどうしても
取れない場合の最小送信機能）。その場合も ADR として記録してから動く。

## 6. 既存ドキュメントとの整合

- **POSITIONING §5** の共存戦略（「統合と施策設計はこちら、配信は MA」）を、本書が
  Own/Integrate/Don't の実装方針に落とした
- **CAPABILITY_MAP §3**「能力拡張＝オントロジー拡張」と一致: Integrate の In は
  すべて取り込みソースの追加、結果データはファクトの追加として実現される
- **ROADMAP M-1/M-3** の解像度を本書が上げた: M-1 は「コネクタ＋結果ファクト」まで
  （配信基盤は Don't）、M-3 は In（CRM インポート）と Out（MA 書き出し）の双方向
- **TECHNICAL_APPEAL** の「マネージドに寄せ、差別化しない基盤を持たない」姿勢の
  事業版が本書である——技術で「持たない」ものを選んだように、事業でも「やらない」
  ものを選ぶ
