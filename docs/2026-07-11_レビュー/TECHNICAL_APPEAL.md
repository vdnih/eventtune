# EventTune — 技術アピール資料

**設計思想・技術選定・汎用 AI 製品との比較優位性**（技術評価者向け）

---

## 1. 何を作ったか — 一言のポジショニング

**「カオスなイベントマーケティングデータをオントロジーに統合し、AI エージェントがその上で
働くマルチテナント SaaS」** である。個別フォロー生成はその最初に実装されたユースケースに
すぎず、今はイベント振り返り・費用対効果分析と並ぶ、複数の中核ユースケースの一つである。

EventTune の技術的な主張は次の一文に集約される:

> **LLM の知能はプロンプトではなく「構造」で統治する。**
> 意味はセマンティックレイヤーに、権限は型に、判断根拠は非 null フィールドに、
> コストは決定論に押し込み、AI には AI にしかできない仕事だけを残す。

```
┌────────────────────────────────────────────────────────┐
│ UI: Next.js 15 (App Router / SSR) — チャット駆動 UI      │
└──────────────┬─────────────────────────────────────────┘
               │ REST / SSE (FastAPI)
┌──────────────▼─────────────────────────────────────────┐
│ Layer 3: MarketingAgent（Google ADK + Gemini, 単一・汎用）│
│   個別/セグメント方式 + HIL / Code Interpreter（隔離サンドボックス）│
├────────────────────────────────────────────────────────┤
│ Layer 2: OSI セマンティックレイヤー（YAML 正典 → Pydantic SSoT）│
│   星座型 5 マスタ + ファクト / appeal_vector 意味空間      │
├────────────────────────────────────────────────────────┤
│ Layer 1: DataIntegrationAgent（AI 抽出 → 決定論変換・名寄せ）│
└────────────────────────────────────────────────────────┘
```

## 2. なぜ汎用 AI（ChatGPT / Gemini / Claude アプリ）では解けないのか

「Gemini に CSV を貼ってメールを書かせればよいのでは」という問いに対する、構造的な回答が
EventTune である。差は機能の量ではなく、**アーキテクチャ上、汎用チャットには原理的に
持てないもの**にある。

なお 2026 年現在、「AI がフォローメールを書く」こと自体は市場のどのカテゴリ
（MA 組み込み AI・イベントリードキャプチャ・AI ライティング）も到達済みであり、
差別化ではない（市場分析は [POSITIONING.md](POSITIONING.md)）。競争軸は文面生成の
**手前**（散在データの意味統合）と**後ろ**（蓄積・学習・監査）に移っており、
以下の 5 点はまさにその手前と後ろを構造で押さえるものである。この構造は模倣コストが
高い——MA 組み込み AI は「整った CRM データ」前提を外せず、イベントツールは
単発イベント単位のアーキテクチャを跨げない。**アーキテクチャの選択がそのまま
市場防御性になっている**ことが、EventTune の技術評価の核心である。

### ① オントロジー接地 — 「状態を持つ AI」vs「毎回コピペ」

汎用チャットの文脈はセッション揮発性で、データの意味構造を持たない。EventTune は
あらゆる入力ファイルを **OSI セマンティックレイヤー**（5 マスタ + ファクトの星座型、
`backend/semantic/osi_event_marketing_v1.yml` が正典）へ正規化してから AI に渡す。
エージェントは「CSV の列」ではなく「Person が Event に参加し Product に関心を持つ」という
意味の上で推論するため、`業種 × イベント × 製品` のようなマルチホップ分析や
イベント横断比較（過去実績との対比）が構造的に可能になる。

### ② コスト構造 — LLM 呼び出しを O(N) から O(K) へ

汎用チャットで 500 人に個別メールを作れば LLM 500 回・レイテンシもコストも線形に爆発する。
EventTune は対象人数で方式を分岐する（**3ゲート＋人数分岐**、ADR-016）。多数向けの
**セグメント方式**（設計原則7）は AI がセグメント軸を設計 → 分類 → **バケット数 K 回だけ**
パターン生成 → 各人への組み立ては**プレースホルダ置換の決定論 Python（LLM ゼロ）**で、
LLM 呼び出しを O(N) から O(K) へ畳み込む。少人数（目安 15 名以下）では、確認負荷を減らす
セグメント化のうまみが小さいため、`generate_individual_deliverables` が1人ずつ実名フル生成する
個別方式を採る。「セグメント化はあくまで多数向けの手段」という原則7の本旨を、人数分岐で
実装に反映した形である。「AI に任せてよいのは意味判断だけ、定型作業をプログラム化しすぎず、
かといって AI に無駄打ちさせない」という責務境界の直接の帰結であり、旧実装（常にセグメント化・
1 バケット 1 回フル生成）の歪みを ADR-016 で是正した記録が残る。

### ③ テナント分離 — 規律ではなく型で保証する

汎用 AI アプリは個人チャットであり、そもそもチームのデータ分離という概念がない。
EventTune はマルチテナント SaaS として、分離を「フィルタの書き忘れがないよう気をつける」
規律ではなく**構造**で担保する（原則6）:

- `SpaceContext`（`backend/space.py`）がテナント文脈を単一の不変オブジェクトに集約し、
  業務コードは `space.col("events")`（スペース前置済み参照）しか手にできない
- 生の `firestore.client()` 呼び出しを業務コードから撤廃（逸脱は grep 一発で検出可能）
- **AI ツールにはさらに狭いケイパビリティ**を与える: `make_tools(db, space)` がスコープ済み
  クライアントを closure で捕捉し、ツールのシグネチャから space_id 自体を排除。
  AI が他テナントを「名指しすることすら不可能」で、**アクセス制御が LLM の判断に一切依存しない**
- クライアント提示の space_id は「主張」であり、毎リクエスト検証済み uid × membership
  ドキュメントから権限を再導出（メンバー削除は次リクエストで即時失効）

### ④ 監査可能性 — Auditable AI を型で強制する

汎用チャットの出力に「なぜ」は残らない。EventTune では**根拠フィールドを Optional に
してはならない**という規約（原則4）を敷く: メール各ブロックの `reason_for_inclusion`、
セグメント割り当ての `reason`、取り込み変換の `TransformDecision.reason` はすべて非 null。
さらに数値分析は LLM に「それっぽい数字」を答えさせず、**AI が生成した Python コードを
実データ上で実行し、コードと結果の両方をチャットに可視化**する（後述の Code Interpreter）。
利用者は AI を信じるのではなく、検証して任せられる。

### ⑤ ブランドガードレール — Static Core & Dynamic Context

汎用 LLM の個別化は「過剰な迎合」（存在しない機能の創作・複数機能の押し売り）と紙一重である。
EventTune はマーケ設計思想（`docs/MARKETING_PHILOSOPHY.md` / ADR-007）として
**AI が書き換えてよい範囲を情報 3 階層で構造的に区切る**: L1/L2（自社の機能・本質価値）は
固定、L3（顧客の悩み＝CEP）だけを AI が生成する。プロンプトのガードレール
（捏造禁止 / 1 メール 1 機能 / ブランド資産維持）＋ HIL 承認ゲートの二段構えで、
「圧倒的なパーソナライズ」と「ブランドの一貫性」を両立させる。

## 3. 設計思想ハイライト

以下はいずれも `docs/ADR.md`（17 本）に意思決定として記録されており、
「なぜそうしたか」「何を捨てたか」まで追跡できる。

### 3.1 OSI セマンティックレイヤー — 意味の単一真実源（ADR-008）

旧実装ではスキーマが Pydantic・プロンプト文・マッパー enum・フロント TS の 4 箇所に
多重定義され、変更のたびにずれた。業界標準 **OSI (Open Semantic Interchange) v1.0** に倣い、
概念モデルを 1 つの YAML に集約し、`ontology.py`（Pydantic）を物理実装として導出する体制へ
再設計。データモデルは **ファクト・コンステレーション（星座型）**: 5 マスタ
（persons / accounts / events / products / contents）は対等で、ファクト
（event_attendances / product_interests …）が結ぶ。初期案の「Event 中心設計」は
運用で誤りと判明し、**ADR に撤回として明記**した上で移行した。

### 3.2 顧客の関心は固定ラベルではなくベクトル空間で（ADR-008 判断5）

当初案の「課題マスタ + 多対多ブリッジ」は、固定ラベルではパターンマッチしかできず
課題に収まらない関心・文脈を表せないため**不採用**。代わりに各マスタへ
`appeal_summary`（監査可能な自然文要約）+ `appeal_vector`（埋め込み）を持たせ、
「この人に合うコンテンツ/製品/イベント」をコサイン類似度（`semantic_search.py`）で引く。
一致根拠は双方の appeal_summary を並べて説明でき、**意味検索でありながら監査可能**。
類似度計算はベクトル DB を持たず決定論 Python の総当たり——スペース単位の小規模データという
前提を明示した上での、意図的な「持たない」選択である。

### 3.3 AI / Python の責務境界 —「意味を変えるか否か」で引く（原則4）

処理の置き場所を「AI が得意か」ではなく **「意味を変えるか否か」** で決める。
抽出・表記正規化は AI、**判断を伴う業務ロジック**（同一性判定・名寄せ・分類確定）は
説明可能な決定論 Python に明文化する。さらに ADR-013 では「取り込み時の業務判定
（温度感の自動分類）」自体を廃止し、観測事実のみを保存してベクトル検索に判断を委ねる
——**判定を良くするのではなく、判定という工程を消す**方向に踏み込んだ。

### 3.4 Code Interpreter — 4 方式を捨てて辿り着いた隔離実行（ADR-009）

自由なデータ分析を「LLM が数字を答える（ハルシネーション）」でも「固定集計ツール
（柔軟性ゼロ）」でもなく、**AI が Python を生成 → Google 管理の Agent Engine サンドボックスで
実行 → コードと結果を UI に可視化**というパターンで実現。ここに至るまでに
`BuiltInCodeExecutor`（FS 非共有で不成立）→ プロセス内実行（マルチテナントで認証情報に
到達しうるため却下）→ `VertexAiCodeExecutor`（基盤自体が廃止予定）→ CodeAct 方式
（モデル挙動と SDK 不整合）の 4 方式を検証して捨てた過程が ADR に時系列で記録されている。
セッションは `VertexAiSessionService` で永続化し、サンドボックスはセッション毎
（`user_id = "{space_id}:{uid}"`）に分離——**LLM 生成コードの実行がテナント境界を
破らないことを基盤レベルで保証**する。

### 3.5 取り込みパイプライン —「観測 → 確定 → 結合 → 導出」（ADR-011 / 013 / 015)

「参加者リストの 1 行」の自然な粒度は Person ではなく**接客の観測**であり、Person マスタは
観測から導出される（DWH の inferred dimension）——という業務ドメインの洞察に基づき、
取り込みを多段パイプラインへ再設計。全エンティティ UUID 主キー + 実在エンティティへの
検索ベース find-or-create に一本化し、名前ハッシュ安定 ID + スタブ生成という
「順序非依存に見せる」脆い仕掛けを全廃した。ADR-015 ではこれを**スペック駆動の統一
8 ステージパイプライン**（確認済みバッチ文脈を全段で共有）へ再建し、CSV/Excel に加え
**PDF/PowerPoint/Word**からのテキスト抽出取り込みにも対応した。AI の使い方も 2 段構え:
バッチ横断理解（`understand_batch`、フルモデル 1 回）+ 行単位並列抽出（軽量モデル
`gemini-3.1-flash-lite`）で、精度とトークンコストを両立させている。

### 3.6 HIL（Human-in-the-Loop）— 自律性と統制の均衡（原則7 / ADR-016）

エージェントは個別対応を自律的にオーケストレーションするが、各ゲートで提案と根拠を提示し
ユーザー承認を得てから確定ツールを呼ぶ。ADR-016 でこれを**3ゲート＋人数分岐**に整理した:
①方針・可否（アプローチ方法の提示と到達可能総数の確認、ここで個別/セグメントを決定）→
②セグメント条件＋人数（多数時のみ）→ ③文面。各ゲートは1応答で確認し（旧実装の「提案」と
「実行します」を別ターンで求める二重確認を撤廃）、承認コストを実運用に合わせて最小化しつつ、
3ゲートいずれもが不可逆操作（`run_assembly` / `generate_individual_deliverables`）の手前に
置かれる安全側の構造は保つ。固定ウィザードでも野放しのエージェントでもない。全成果物
（Segment / 割り当て根拠 / パターン / 個別成果物）は Firestore に残り、事後監査可能。

### 3.7 インフラ — IaC の責務分割とキーレス CI/CD（ADR-012）

「全部 Terraform」ではなく責務で割る: GCP の「箱」（Cloud Run / IAM / WIF / Firestore DB）は
Terraform、ルールは Firebase CLI、フロントは App Hosting の push 連動、バックエンドは
GitHub Actions（WIF キーレス、SA 鍵なし）。プロバイダのリソースが実際の使い方を
モデル化していない箇所（Agent Engine / Auth）は無理に IaC 化しない判断も記録済み。
課金は機能単位メトリクスを取らず**原価直結の生実績（トークン・実行時間）のみ計測し、
読み取り時にクレジット換算**する（レート改定が遡及一貫する）設計。

## 4. 技術選定

| レイヤー | 選定 | 理由（要点） |
|---|---|---|
| フロントエンド | Next.js 15 (App Router / SSR) + Tailwind | 認証ゲート・ユーザー毎データに SSR が有利。App Hosting が直接サポート（ADR-005） |
| バックエンド | Python 3.12 + FastAPI | ADK / SSE ストリーミングとの親和性。Cloud Run シングルサービスで運用単純化（ADR-002） |
| AI オーケストレーション | Google ADK + Gemini（Vertex AI）。推論・生成は `gemini-3.5-flash` 主体、高ボリュームの行単位抽出のみ `gemini-3.1-flash-lite` | Agent + Tools 型。用途別モデル選択によるコスト設計（行単位並列抽出は軽量モデル・K 回のみのパターン生成） |
| 埋め込み | gemini-embedding-001 | appeal_vector の生成。消費側は決定論 Python |
| コード実行 | Agent Engine Code Execution Sandbox | 唯一の本物の隔離境界。Cloud Run と分離するデカップリングは GCP 公式パターン（ADR-009） |
| DB / 認証 | Firebase（Firestore / Auth / Storage）| スペース前置パス構造がテナント分離と噛み合う。ルールでクライアント書込を統制 |
| IaC / CI | Terraform + GitHub Actions (WIF) + App Hosting | 責務分割・キーレス（ADR-012） |

一貫しているのは **「マネージドに寄せ、差別化しない基盤を持たない」** 姿勢である
（ベクトル DB なし・ワーカーキューなし・モノレポツールなし——いずれも導入しない理由と
再評価トリガーが ADR に明記されている）。

## 5. エンジニアリング文化 — このリポジトリ自体が成果物

ハッカソン評価の観点で最後に強調したいのは、コードだけでなく**意思決定の運用**である。

- **ADR 17 本**: 採用理由だけでなく「捨てた方式」「撤回した原則」（Event-Centric 撤回、
  Challenge マスタ撤回、EngagementLevel 分類の廃止、由来遡及機能の撤去〔ADR-017〕）を残す。
  失敗・撤回の記録が次の設計を守る。
- **doc⇔impl 監査を工程化**（ADR-010): 大型 PR 後に「思想が WRITE 側だけ実装されていた」
  「死蔵 API が残っていた」といった乖離を体系的に棚卸しし、「実装で得た学びは実装が正、
  未到達の思想は実装を追いつかせ、撤回概念は実装からも退役」という原則で是正した記録がある。
- **思想 → 実装のトレーサビリティ**: マーケ思想（MARKETING_PHILOSOPHY）→ システム原則
  （PHILOSOPHY_AND_NAMING）→ ADR → プロンプト・コードの各層が相互参照され、
  「なぜこのプロンプト行があるのか」を思想まで遡れる。
- **テスト**: 取り込みパイプライン・名寄せ・マッパー・エージェントのバックエンドテスト
  （`backend/test_*.py`）+ GitHub Actions CI。
- **能力境界の明文化**: 「できないこと」を未実装と**意図的な不許可**（テナント越境・
  承認なし確定・L1/L2 の創作・根拠なし出力）に区別して定義している
  （[CAPABILITY_MAP.md](CAPABILITY_MAP.md)）。後者は欠落ではなく設計思想の証拠——
  「何をさせないか」を構造で保証できて初めて、エージェントに自律性を渡せる。

**AI プロダクトの信頼性は、モデルの賢さではなく、賢さを統治する構造から生まれる**——
EventTune はその構造を、データモデル・権限・コスト・監査・ドキュメントの
5 面すべてで実装した試みである。
