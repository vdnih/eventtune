# ケイパビリティマップ — できること・できないこと・向き不向き

EventTune は「メール生成ツール」ではなく、オントロジーの上で働く汎用マーケティング
エージェントである。理想は「なんでもできる」だが、実際には明確な境界がある。
本ドキュメントはその境界を **場当たりのリストではなく、原理から** 整理する。
利用者への期待値設定、営業トーク、オンボーディング設計の基礎資料を意図している。

---

## 1. 境界を決める3つの問い

「この仕事を頼めるか？」は、次の3つの問いでほぼ機械的に判定できる。

| # | 問い | 根拠となる設計原則 |
|---|---|---|
| Q1 | **その仕事に必要な「意味」は、オントロジーに載っているか？** | 原則2: あらゆるデータをオントロジーに通す。エージェントは生データや外部世界ではなく、取り込まれた意味の上でだけ働く |
| Q2 | **その仕事は「意味の判断」か「定型作業」か？** | 責務境界: 意味判断（文章・分類・軸設計）は AI、定型作業（組み立て・集計実行）は決定論 Python。どちらかに落ちれば実行可能 |
| Q3 | **確定的な影響を持つ操作か？** | 原則7 (HIL): 全件確定などの不可逆操作は人間の承認ゲートを必ず通る。承認なしには進まない |

Q1 が No なら「できない」か「まず取り込みが必要」。Q1 が Yes なら大抵できる。
Q3 に該当する操作は「できるが、勝手にはやらない」。

## 2. ケイパビリティ・スペクトラム

### ◎ 得意 — 設計の中心（このために作られている）

| できること | 実装根拠 |
|---|---|
| **イベント後の個別フォロー成果物を全員分、一括生成** — AI が方針・可否を提示し、対象人数で分岐: 少人数は1通ずつ実名フル生成（個別方式）、多数はセグメント軸を設計→分類→バケット別パターン生成→決定論組み立て（セグメント方式）。数百人規模でも高速・低コスト | 3ゲート＋人数分岐（原則7 / ADR-016）: 個別方式 `generate_individual_deliverables` / セグメント方式 `define_segment` / `assign_segment` / `generate_patterns` / `run_assembly` |
| **カオスなファイル群の意味統合** — 列名・表記がバラバラの CSV/Excel/テキスト、PDF/PowerPoint/Word（参加者リスト・イベント概要・アンケート・KPI・費用）をまとめてドロップ→自動で名寄せ・リンク解決 | DataIntegrationAgent（ADR-011/013/015 の統一パイプライン、find-or-create 名寄せ） |
| **イベント振り返り・ROI・分布の自由分析** — 「業種分布は？」「費用対効果は前回比で？」に、AI が Python を書き実データで計算。コードも結果も見える | Code Interpreter（ADR-009）: `get_space_data` + `run_python_code` |
| **「この人に合うもの」の意味検索** — 参加者の関心・悩みに意味的に近いコンテンツ・製品・イベントを、根拠つきで引き当てる | `find_relevant_for_person`（appeal_vector コサイン近接、ADR-008） |
| **イベント横断の蓄積・比較** — 過去イベントの実績（参加・KPI・費用）を跨いだ比較・学習 | 原則5 Historical Intelligence（星座型データモデル） |
| **すべての判断の根拠提示** — 分類理由・文面の採用理由・取り込み解釈の判断（`TransformDecision`）が記録される | Auditable AI（原則4、非 null reason）。※ エンティティ単位で由来を遡る UI/API は ADR-017 でいったん撤去（再設計待ち） |

### ○ できる — 汎用性の範囲内（頼めば動くが、磨き込みは発展途上）

| できること | 注記 |
|---|---|
| **アドホックな質問・分析** — 定型メニューにない依頼も、データが取り込まれていれば AI がツールを組み合わせて対応 | チャット＋ツール方式（原則3）の恩恵。分析の自由度はサンドボックス内 Python の範囲 |
| **メール以外のフォーマット** — トークスクリプト・提案書 | `Deliverable.format` は配線済み。ただし生成プロンプトは EMAIL 主体で品質差がある（ADR-010 将来課題） |
| **戦略レポートの作成・保存** | `save_report`。チームで後から参照可能 |
| **取り込み済みデータの閲覧・エクスポート** — `/data` 画面での閲覧（テーブルの join・集計・AI サマリ・列管理）、成果物の CSV 出力 | `/api/data/*` + `/data` 画面（#33 で刷新） |
| **セグメントの軌道修正** — 軸変更・対象絞り込み・トーン変更を会話で指示 | HIL の各ゲートでやり直し可能 |

### △ 向いていない — 動くが本領ではない（他のツールを使うべき）

| 頼めるが勧めないこと | 理由 |
|---|---|
| **イベント・顧客データと無関係な汎用コピーライティング**（ブログ記事・広告コピー・SNS 投稿等） | オントラジー接地が価値の源泉であり、接地なしなら汎用 AI チャットや AI ライティングツールの方が適する（Q1 が実質 No） |
| **リアルタイムの外部情報が必要な仕事**（競合調査・最新ニュース反映・相手企業の Web リサーチ） | エージェントは Web 検索ツールを持たない。世界の知識はモデルの学習範囲とオントロジー内に限られる |
| **超大規模リストの処理**（数万〜規模） | 意味検索は決定論総当たり O(N) 前提（ADR-008）。スペース単位の小規模データという設計前提の外 |
| **厳密なレイアウトを持つ成果物**（デザインされた LP・スライド・画像） | 成果物はテキスト構造（ブロック）であり、ビジュアル生成の機構を持たない |

### ✕ できない — 構造的な境界

「できない」には性質の異なる2種類があり、**この区別が重要**である。

#### (a) まだ実装していない（ロードマップの対象 → [ROADMAP_PROPOSAL.md](ROADMAP_PROPOSAL.md)）

| できないこと | 現状の代替 / 予定 |
|---|---|
| メールの**送信・配信** | CSV エクスポートして既存の配信ツールへ（送信連携は M-1） |
| 開封・返信などの**効果測定** | ループ未接続（M-1 で解消予定） |
| CRM / MA からの**直接インポート・外部 SaaS 連携** | ファイルエクスポート経由（M-3） |
| 分析結果の**チャート表示** | 数値・表のみ（S-4） |
| エンティティ単位の**由来遡及 UI**（どのファイル/行から確定したか） | ADR-017 で断片化した3経路ごと撤去。`source_job_id` / `TransformDecision` は残るが逆引きは非提供。再設計は要望具体化時（S-1） |

#### (b) 意図的にできなくしてある（設計上の不許可 — 直す予定のない「できない」）

| できないこと | なぜ意図的か |
|---|---|
| **他スペース（テナント）のデータへの一切のアクセス** | AI ツールは自スペースを「名指しすることすら不可能」な closure 束縛（原則6）。これが SaaS としての信頼の土台 |
| **承認なしの全件確定** — `run_assembly` 等をユーザー承認なしに実行 | HIL（原則7）。不可逆な確定は人間の判断を必ず経る |
| **自社の機能・価値（L1/L2）の創作** — 存在しない機能を語る、複数機能を押し売りする | Static Core & Dynamic Context（ADR-007）。AI が書き換えてよいのは顧客文脈（L3）だけ |
| **根拠なしの出力** — 「なぜそうしたか」を持たない分類・文面 | Auditable AI（原則4）。根拠フィールドは Optional にできない |
| **オントロジーを迂回した生データ利用** — PDF や CSV を「そのまま」プロンプトに入れて処理 | 原則2。必ず取り込み（意味化）を経由する。即答性より一貫性・監査性を取る |

(b) は制約ではなく **プロダクトの約束** である。「AI が勝手にブランドを壊さない」
「勝手に送らない」「説明できないことをしない」は、この「できない」によって保証されている。

## 3. 含意 — 「なんでもできる」の正体

整理すると、EventTune の汎用性は次の一文に集約される:

> **「オントロジーに載った意味の上でなら、何でもできる」**

したがって、できることの境界を広げる正攻法は、エージェントに機能を足すことではなく
**オントロジーを広げること**である。実例がすでにある:

- 費用分析がしたい → `cost_items` ファクトを追加（実装済み。正典 YAML・SEMANTIC_LAYER と同期済み）
- 施策の効果で学習したい → 送信結果のファクト（例: `deliverable_outcomes`）を追加（ROADMAP M-1）
- ファクトは「今後さらに増える前提」で設計済み（星座型、ADR-008）

逆に言えば、**オントロジーに載らない仕事（外部リアルタイム情報・ビジュアル制作・
接地なしの汎用ライティング）は、今後も本領にはならない**。そこは汎用 AI や専門ツールに
任せ、EventTune は「自社のイベントマーケデータの上で働く、判断の見える
エージェント」であり続けるのが、能力境界から導かれる自然な戦略である
（市場的な裏付けは [POSITIONING.md](POSITIONING.md)）。
