# 今後の機能追加・改善・拡張の提言

本ドキュメントは、利用者向け訴求（[PRODUCT_INTRODUCTION.md](PRODUCT_INTRODUCTION.md)）・
技術訴求（[TECHNICAL_APPEAL.md](TECHNICAL_APPEAL.md)）・市場ポジショニング
（[POSITIONING.md](POSITIONING.md)）・能力境界（[CAPABILITY_MAP.md](CAPABILITY_MAP.md)）の
検討を踏まえ、**「どの価値を強化するために、何を、どの順で作るべきか」** を提言する。

## 提言の軸 — 検討から導かれる投資基準

各資料を書いて明確になったのは、本プロダクトの価値が次の2軸に立っていることである。

- **利用者価値の核**: 「作業が消え判断だけが残る」「AI の判断が見える」「使うほど過去が
  未来を照らす」
- **技術価値の核**: 「LLM を構造で統治する」（オントロジー接地・監査可能性・コスト設計・
  テナント分離）

したがってロードマップの判断基準は「機能が増えるか」ではなく、次の3点とする:
**①価値ループ（データ→施策→結果→学習）が閉じるか、②統治の構造が保たれるか、
③市場ポジション（意味統合 × 監査可能性 × 蓄積）を守り・広げるか**。

③を補足する市場的前提が2つある（POSITIONING §1-2）。第一に、**「AI がフォローメールを
書く」ことは 2026 年時点で MA 組み込み AI もイベントツールも到達済み**であり、文面生成への
追加投資は差別化を生まない。第二に、能力境界の分析（CAPABILITY_MAP §3）が示した通り、
**本プロダクトの能力拡張の正攻法は機能追加ではなくオントロジー拡張（ファクト追加）**である。
この2つが、以下の優先順位——文面生成そのものより「根拠の露出」「結果の蓄積」「統合の入口」
への投資——を裏付ける。以下、短期→中期→長期の3ホライズンで提言する。

---

## 短期 — 体験を「閉じる」（実装済み機能の最後の 1 マイル）

すでにバックエンドに存在するが UI・体験に到達していない価値を露出させる。
投資対効果が最も高い領域。

### S-1. 加工処理レポートの UI 化

- **現状**: 取り込みの変換根拠（`TransformDecision` / スキップ理由 / 横断伝播）は
  `IntegrationJob` に完全に記録され API もあるが、UI が「将来実装」のまま。
- **提言**: Explorer に「この取り込みで何が起きたか」ビューを追加する。
- **なぜ今か**: Auditable AI は本プロダクトの第一の差別化訴求だが、現状ユーザーに見えるのは
  成果物側の根拠のみ。**取り込み側の透明性はデモで最も「おっ」と言わせられる未露出資産**である。
  文面生成がコモディティ化した市場（POSITIONING §1）では、競合が持たない「根拠」の露出こそが
  見た目の差別化になる。

### S-2. パターン編集・HIL 介入の UI 復活

- **現状**: ADR-010 で segments router を削除したため、生成パターンへの人間の介入窓口は
  チャット内の会話のみ。マーケターが文面を直接微修正する UI がない。
- **提言**: `data.router` 側に読み取りを足し、パターン（`DeliverablePattern`）の閲覧・編集
  UI を再設計する（ADR-010 が予告した通りの再設計）。
- **なぜ今か**: HIL は「人が承認する」訴求の根幹。承認はできるのに**微修正ができない**のは
  体験の穴で、実利用では「9割良いがこの一文だけ直したい」が必ず起きる。

### S-3. 生成品質の評価ハーネス（eval）

- **現状**: ガードレール（捏造禁止・1機能1CEP・トーン維持）はプロンプトのソフト強制のみで、
  回帰を検知する仕組みがない。モデル更新・プロンプト変更が品質を壊しても気づけない。
- **提言**: サンプルデータ＋既知の期待特性（「Content に無い機能を語らない」「1通に機能は
  1つ」等）を LLM-as-judge で検査する eval スイートを CI に追加する。
- **なぜ今か**: 「ガードレールがある」という技術訴求を**検証可能な主張**に変える。
  ADR-007 が明示したトレードオフ（ソフト強制の限界）への正面回答になる。

### S-4. チャート出力（分析の可視化）

- **現状**: Code Interpreter は数値のみ。matplotlib 画像の受け取りは ADR-009 の将来課題。
- **提言**: サンドボックスの出力ファイル受け取りを実装し、分析結果をグラフでチャットに表示。
- **なぜ今か**: 「AI が実行したコード」パネルという既存の可視化資産に直結し、
  振り返り分析の説得力（デモ映え・実用の双方）が大きく上がる。

## 中期 — 価値ループを「閉じる」（プロダクトの本丸）

### M-1. 送信連携と効果測定 — Historical Intelligence を実データで駆動する

- **現状**: 成果物は CSV エクスポートで終わり、開封・返信・商談化が戻ってこない。
  原則5「過去が未来を照らす」は、イベント実績（KPI・費用）では機能するが、
  **施策の結果**については学習ループが開いたままである。
- **提言**: SendGrid 等での送信連携（または MA ツールへの書き出し + 結果の取り込み口）を作り、
  開封・返信を `Deliverable` に紐づくファクト（例: `deliverable_outcomes`）として蓄積する。
  **スコープの線引き**: 作るのは配信ツールへの**コネクタ**と**結果ファクトの取り込み**まで。
  配信基盤そのもの（SMTP・到達性・購読解除管理）は戦略的に実施しない
  （[STRATEGIC_SCOPE.md](STRATEGIC_SCOPE.md) §5）。
- **接続点**: ファクトは「今後増える前提」で設計済み（星座型）。結果ファクト
  （例: `deliverable_outcomes`）の追加は、CAPABILITY_MAP §3 の「能力拡張＝オントロジー拡張」の
  教科書的適用である。結果が入れば「前回この訴求のセグメントは返信率が高かった」を
  セグメント軸設計の入力にでき、**本プロダクトにしかできない自己改善ループ**が完成する。
- **市場文脈**: POSITIONING §5 が示す最大の脅威は MA 組み込み AI の進化だが、施策結果まで
  含めたイベント横断の学習ループは、単発文面生成（Cvent 等が到達済み）と違い模倣に
  データ蓄積年数を要する。**M-1 はポジショニング上の最重要防御投資**でもある。

### M-2. L1/L2 Static Core の構造化マスターと Brand Context 注入

- **現状**: ブランドの「不変のコア」の事実上の真実源は Content / Product コレクションで、
  L1（プラットフォーム指針）・トーン規定・禁止ワードの構造化マスターは未実装
  （ADR-007 の明示的な将来課題）。
- **提言**: `ontology.py` に BrandContext 系モデルを追加し、`make_tools` でプロンプトへ
  注入する機構＋設定 UI を作る。
- **接続点**: ADR-004 が指摘した通り **Brand Context は SaaS の課金根拠そのもの**
  （設定に投資するほど出力品質が上がる）。ガードレール③（DBA 維持）がデータ駆動になり、
  eval（S-3）の検査対象にもなる。

### M-3. CRM / MA 連携（HubSpot・Salesforce インポート）

- **提言**: ハウスリストの CRM 直接インポート。取り込みは「観測→確定→結合→導出」
  パイプライン（ADR-011/013）に API ソースを1つ足すだけで、find-or-create の名寄せ機構は
  そのまま流用できる。
- **なぜ**: 「ファイルをドロップ」の手前にある実務（CRM からのエクスポート作業）を消し、
  導入摩擦を下げる。エンタープライズ商談の前提条件でもある。さらに POSITIONING §6 の
  対 MA トーク「統合と施策設計はこちら、配信は MA と併用」を体験として成立させるには、
  取り込みだけでなく **MA への書き出し**（セグメント・成果物の連携）も本項に含める。
  共存ポジションは語るだけでなく配線されて初めて防御になる。連携の方向（In/Out）と
  受け渡すデータの設計は [STRATEGIC_SCOPE.md](STRATEGIC_SCOPE.md) §4 に定義した。

### M-4. マルチフォーマット成果物の本格化

- **現状**: `Deliverable.format`（EMAIL / TALK_SCRIPT / PROPOSAL）は配線済みだが、
  生成プロンプトは EMAIL 主体（ADR-010 の将来課題）。
- **提言**: インサイドセールス向けトークスクリプトを次のフォーマットとして精緻化する。
  同じセグメント・同じ根拠から「メール＋電話スクリプト」が同時に出る体験は、
  「メール生成ツールではない」というポジショニングの最短の証明になる。

### M-5. 課金の接続（Stripe）

- **現状**: メータリング（生実績計測）とクレジット換算・プラン定義（`metering.py` /
  `plans.py`）は実装済み。決済とプランゲーティングが未接続。
- **提言**: Stripe サブスクリプション連携と、クレジット超過時の挙動（ソフトリミット→通知→
  ハードリミット）を実装する。設計上の難所は既に解かれており、残りは配線である。

## 長期 — プラットフォームへ「広げる」

### L-1. パブリック領域 / AEO への展開

MARKETING_PHILOSOPHY §3① が思想として定義済みの未実装領域。オントロジー（L1/L2/L3 の
構造化情報）から、AI クローラーに最適化された構造化 Web コンテンツ（機能別ページ・FAQ）を
生成する。**同じ Static Core から私信（メール）と公開情報（Web）の両方が矛盾なく出る**ことは、
思想の完成形であり、他社が模倣しにくい構造的な差別化になる。

### L-2. スケール対応 — 明文化済みトリガーの監視

ADR には「持たない」選択の再評価トリガーが記録済みである。ロードマップとしては
**新機能ではなくトリガー監視**を提言する:

| 対象 | 現状の選択 | 移行トリガー（ADR 記載） |
|---|---|---|
| ベクトル類似度 | 決定論総当たり O(N) | スペースの大規模化 → 近似最近傍 / 事前計算（ADR-010） |
| ジョブ実行 | Cloud Run BackgroundTask | スケール限界 / 長時間ジョブ必須化 → Cloud Tasks or Agent Engine（ADR-002/009） |
| 環境 | 単一プロジェクト | dev/staging 需要 → `var.project_id` で複製（ADR-012） |

### L-3. エンタープライズ要件

効果測定（M-1）と CRM 連携（M-3）が揃った後の展開として、監査ログの外部出力・
データレジデンシー（テナント別プロジェクト分離）・SOC 2 対応。テナント分離が既に
「構造」で実装されているため、コンプライアンス文書化のコストは相対的に低い。

## 優先順位の総括

| 順位 | 項目 | 理由 |
|---|---|---|
| 1 | S-1 / S-2（透明性 UI・HIL 介入 UI） | 実装済み価値の露出。最小コストで訴求が最大化 |
| 2 | S-3（eval） | 品質主張を検証可能にする。以降の全変更の安全網 |
| 3 | M-1（送信・効果測定） | 唯一開いている価値ループを閉じる。構造的優位の完成 |
| 4 | M-2（Brand Context） | 課金根拠と品質の同時強化 |
| 5 | M-5 → M-3 → M-4 | 商用化の配線 → 導入摩擦の除去 → ポジショニング証明 |

**やらないことの提言**も添える: ベクトル DB 導入・ワーカーキュー分離・マルチエージェント化は、
ADR に記録されたトリガーが発火するまで着手しない。「持たない」選択の一貫性こそが
本プロダクトの技術的な強みであり、それを守ること自体がロードマップの一部である。
